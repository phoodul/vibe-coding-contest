# Claude Code 토큰 관리 전략

> 7일 대회 기간 동안 토큰을 전략적으로 사용하여 마지막 날까지 개발 가능하도록 관리

---

## 1. 서브에이전트 모델 라우팅

모든 작업에 Opus를 쓸 필요 없다. 작업 성격에 따라 모델을 배정한다.

| 에이전트 | 모델 | 이유 |
|---------|------|------|
| **Architect** | Opus (기본) | 설계 판단, 트레이드오프 분석 — 사고력 필요 |
| **Implementer** | Opus (기본) | 핵심 로직 구현 — 코드 품질 중요 |
| **Researcher** | Opus | 정확한 리서치와 판단을 위해 Opus 사용 |
| **Reviewer** | Opus | 보안 점검 + 품질 판단 — 정확도 우선 |
| **Designer** | Opus | 디자인 품질과 Vibe Factor 판단 — 품질 우선 |

**현재 전략:** Max $110 플랜. 품질 최우선이므로 모든 에이전트 Opus 사용. 토큰 절약은 "기능별 새 대화 시작"과 "불필요한 파일 재읽기 방지"로만 관리. 한도 도달 시 사용자에게 사전 알림 후 상향 결정.

## 2. 대화 관리 전략

### 기능별 새 대화 시작 (가장 중요)

긴 대화 = 매 턴마다 전체 컨텍스트 재전송 = 토큰 폭발.

```
❌ 하나의 대화에서 모든 기능 구현 (컨텍스트 오염 + 토큰 낭비)

✅ 기능별 새 대화 시작:
   대화 1: Auth 구현 → 완료 후 종료
   대화 2: Dashboard CRUD → 완료 후 종료  
   대화 3: AI 에이전트 연동 → 완료 후 종료
```

### 새 대화를 시작해야 하는 시점
- 현재 기능이 완료되어 커밋한 후
- 대화가 20턴 이상 길어졌을 때
- 전혀 다른 영역의 작업으로 전환할 때
- "컨텍스트가 압축되었습니다" 알림이 뜨면 즉시

### 새 대화 시작 시 컨텍스트 주입 템플릿
```
현재 상태: [어떤 기능이 완성된 상태인지]
다음 목표: [이번 대화에서 구현할 기능]
관련 파일: [참고할 파일 경로 1~3개]
```

## 3. 토큰 효율적 프롬프트 패턴

### 파일 읽기 최적화
```
❌ "이 프로젝트의 모든 코드를 읽어줘"
✅ "app/dashboard/page.tsx의 30~60줄만 읽어줘"

❌ "프로젝트 구조를 분석해줘" (Agent Explore 호출 → 대량 파일 읽기)
✅ "app/ 폴더의 page.tsx 파일들만 찾아줘" (Glob으로 1회)
```

### 코드 수정 최적화
```
❌ "이 파일을 전부 다시 작성해줘" (Write → 전체 파일 출력)
✅ "handleSubmit 함수만 수정해줘" (Edit → 변경 부분만)
```

### 질문 최적화
```
❌ "이 코드가 뭐하는 건지 자세히 설명해줘" (긴 설명 생성)
✅ "이 함수의 반환값이 뭐야?" (짧은 답변)
```

## 4. 일별 토큰 예산 배분 (7일 기준)

| Day | 작업 | 토큰 비중 | 전략 |
|-----|------|----------|------|
| 1 | 기획 + 설계 | 10% | Architect(Opus) 1회, 나머지는 직접 작성 |
| 2-3 | 핵심 개발 | 35% | Implementer(Opus)에 집중, 기능별 대화 분리 |
| 4 | 통합 + 안정화 | 15% | Reviewer(Sonnet)으로 리뷰, 짧은 디버깅 대화 |
| 5-6 | UI 광택 + Wow | 25% | Designer(Sonnet) + 짧은 수정 대화 |
| 7 | 제출물 완성 | 15% | 리포트 작성, README 정리 |

**핵심:** Day 2-3에 가장 많이 사용하되, Day 7 제출용 토큰을 반드시 남겨둔다.

## 5. 토큰 낭비 안티패턴

| 안티패턴 | 토큰 낭비량 | 대안 |
|---------|-----------|------|
| 긴 대화에서 계속 작업 | 매 턴 컨텍스트 비용 증가 | 기능 완료 후 새 대화 |
| "모든 파일 읽어줘" | 수천~수만 토큰 | Glob/Grep으로 정확히 찾기 |
| 전체 파일 Write | 파일 크기만큼 출력 | Edit으로 변경분만 |
| 불필요한 설명 요청 | 수백 토큰 응답 | 코드로 직접 보기 |
| 같은 에러 반복 디버깅 | 매 시도마다 전체 컨텍스트 | 에러 메시지 복사 → 새 대화 |
| Explore 에이전트 남용 | 수십 파일 읽기 | Glob/Grep 직접 사용 |

## 6. 긴급 상황: 토큰 부족 시 대응

1. **Sonnet으로 전환:** 메인 대화도 `/model sonnet`으로 전환
2. **서브에이전트 최소화:** 직접 Glob/Grep/Read로 작업
3. **코드 생성 최소화:** 구조만 잡고 세부 구현은 직접 타이핑
4. **리포트에 이 상황 기록:** "토큰 한계에 도달하여 직접 코딩으로 전환" — 오히려 AI 활용 리포트에서 좋은 사례

---

## 7. /compact 명령어 활용

대화가 길어지면 `/compact`로 수동 컨텍스트 압축 가능:
- 핵심 결정사항과 현재 상태만 유지
- 이전 코드 탐색 내용은 제거
- 압축 후에도 작업 계속 가능
