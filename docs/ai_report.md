# AI 활용 리포트

## 1. 프로젝트 개요
- **프로젝트명:** MindPalace (마인드팰리스)
- **한 줄 설명:** AI 기억의 궁전 학습 + 교사 AI 어시스턴트
- **해결하려는 문제:** 학생의 개인화 학습 부재와 단순 반복 암기, 교사의 수행평가 피드백/생기부 업무 과부하
- **대상 사용자:** 고등학생 (학습), 교사 (수행평가 피드백 + 생기부)

## 2. 사용한 AI 도구 및 역할 분담

| AI 도구 | 활용 영역 | 선택 이유 |
|---------|----------|----------|
| Claude Code CLI (Opus 4.6) | 전체 개발 — 아키텍처 설계, 코드 구현, 디버깅, 문서 작성 | 1M 컨텍스트로 프로젝트 전체를 이해하며 코딩, MCP 도구 직접 연동 |

## 3. 프롬프트 전략

### 3-1. 역할 부여 (서브에이전트 시스템)
7개의 전문 서브에이전트를 사전 구축하여, 하나의 AI를 역할별 팀으로 분리했다:
- **Researcher**: 최신 자료 조사 전담 (코드 작성 금지)
- **Architect**: PRD, 스키마 설계 전담 (Bash 실행 금지)
- **Implementer**: 코드 구현 전담
- **Reviewer**: 코드 리뷰 전담 (코드 수정 금지)
- **Designer**: UI/UX 피드백 전담
- **Debugger**: 에러 진단-수정-재검증 루프 전담
- **Improver**: 코드 개선 + Researcher 연동 의사결정 (신규 구축)

**핵심 설계 원칙**: 각 에이전트의 도구 접근 권한을 제한하여, 역할 간 간섭을 방지했다. 예를 들어 Reviewer는 Edit 도구가 없어 "리뷰어가 직접 고치는" 안티패턴이 불가능하다.

### 3-2. 컨텍스트 관리
- CLAUDE.md에 프로젝트 규칙을 사전 정의하여 모든 대화에서 일관된 행동 유도
- implementation_plan.md를 AI와 공유하여 계획 정렬 후 구현
- Memory 시스템으로 대화 간 연속성 유지 (사용자 프로필, 대회 정보, 기술 결정 등)

### 3-3. 자율 진행 패턴
대회 특성상 사용자(개발자)가 업무/수면 중에도 AI가 자율적으로 작업해야 했다. "전권 위임" 패턴을 적용:
- 사용자가 방향만 설정하고 "자율 진행해줘"로 위임
- AI가 태스크 목록을 자체 관리하며 우선순위 결정
- 빌드 성공 → 커밋 → Push → Vercel 자동 배포까지 독립 실행
- 에러 발생 시 자체 진단 → 수정 → 재검증 루프

## 4. AI로 해결한 핵심 문제

### 문제 1: 기억의 궁전 방법론의 디지털 구현
- **상황:** 기억의 궁전(Method of Loci)은 2,500년 역사의 기억법이지만, 디지털 도구로 제대로 구현된 사례가 없었다. 특히 "단원마다 정보량이 다를 때 장소를 어떻게 유연하게 확장하는가"가 핵심 과제였다.
- **AI 활용:** Researcher 에이전트에게 "기억의 궁전 실전 활용 방법론"을 조사하도록 위임. 세계 기억력 챔피언들의 기법(VAKOG 다감각 인코딩, PAO 시스템, 소품 레이어링)을 조사하여 보고서 생성.
- **내 판단:** Researcher가 제시한 "소품 레이어링" 전략(하나의 구역을 위/아래/안/옆/인물로 세분화)을 채택. VR 기반 접근은 7일 제약상 배제.
- **결과:** `/api/palace/expand` 엔드포인트로 노드 수 > 구역 수일 때 AI가 장소 특성에 맞는 소품을 동적 생성. 프롬프트에 VAKOG + 6대 강화 기법을 필수 포함하도록 설계.

### 문제 2: AI 마인드맵이 성취기준과 무관한 내용 생성
- **상황:** 초기 마인드맵 API가 "세포와 세포 분열" 단원을 요청해도 "항상성과 건강" 등 다른 단원의 내용을 생성하는 문제 발생.
- **AI 활용:** 프롬프트를 3차에 걸쳐 반복 개선. temperature를 0.3으로 낮추고, "성취기준에 없는 내용은 절대 포함하지 마세요" 지시를 추가.
- **내 판단:** 프롬프트 엔지니어링만으로는 한계가 있어, 스키마 구조 자체를 변경. `subNodes` 필드를 추가하여 교과서 목차 구조(단원→소단원→세부주제)를 강제.
- **결과:** 마인드맵이 교과서 구조를 정확히 반영하게 되었고, subNodes의 detail에 시험에 나올 수준의 구체적 내용(정의, 과정, 예시)이 포함됨.

### 문제 3: Vercel AI SDK v6 호환성 문제
- **상황:** 교사 도구에서 `useCompletion` + `toDataStreamResponse` 조합이 작동하지 않음. 피드백이 생성되지만 UI에 표시되지 않는 문제.
- **AI 활용:** API를 직접 curl로 테스트하여 서버는 정상 응답하지만 클라이언트가 파싱하지 못하는 것을 확인. SDK 버전(v6)에서 `toDataStreamResponse`가 `toTextStreamResponse`로 변경된 것을 발견.
- **내 판단:** `useCompletion` 훅 대신 직접 `fetch` + `ReadableStream` + `TextDecoder`로 구현. SDK 의존성을 줄이고 더 안정적인 방식 선택.
- **결과:** 교사 도구에서 실시간 스트리밍이 정상 작동. 피드백이 글자 단위로 나타나는 타이핑 효과 구현.

### 문제 4: 사전 준비 → 주제 대응의 전략적 분리
- **상황:** 대회 주제 공개(04.06) 전에 "어떤 주제가 나와도 대응 가능한 상태"를 만들어야 했다.
- **AI 활용:** Gemini Pro와 Claude Opus에게 독립적으로 우승 전략을 요청하여 교차 분석. "기획에 50-73% 시간 투자", "하나의 Wow Feature에 올인" 등 공통 패턴 도출.
- **내 판단:** 코드 0줄 상태에서 5개 서브에이전트, 7개 스킬, 4개 Hook, 6개 MCP 서버, 문서 템플릿을 사전 구축. 주제 공개 후 Day 1에 기획만 하면 바로 구현 가능한 상태를 만듬.
- **결과:** 주제 공개 당일 밤에 PRD → Implementation Plan → MVP 전체 구현 → 배포까지 완료. 사전 준비가 없었다면 Day 2-3까지 인프라에 시간을 소모했을 것.

## 5. 나 ↔ AI 워크플로우 (솔로 개발 사이클)

### 개발 사이클
```
[사용자] 방향 설정 + 핵심 판단
    ↓
[Researcher] 리서치 → 근거 자료 수집
    ↓
[사용자 + Architect] PRD + implementation_plan 작성
    ↓
[사용자] 승인
    ↓
[Claude Code] 자율 구현 (빌드 → 테스트 → 커밋 → Push → 배포)
    ↓
[Playwright] 시각적 검증 (스크린샷 피드백)
    ↓
[사용자] 피드백 → 다음 사이클
```

### 효과적이었던 패턴
1. **"전권 위임" 자율 진행**: 사용자가 수면/업무 중에도 AI가 독립적으로 기능 구현, 테스트, 배포까지 수행. 11개 커밋을 자율적으로 생성.
2. **병렬 리서치**: 한국 교육 현장 페인포인트 + 글로벌 AI 교육 사례를 2개의 Researcher 에이전트가 동시에 조사. 30분 만에 양쪽 보고서 완성.
3. **Playwright 시각적 피드백 루프**: 코드 작성 → Playwright 스크린샷 → 시각적 확인 → 즉시 수정. UI 버그를 코드 레벨이 아닌 시각적 레벨에서 즉시 발견.
4. **MCP 직접 연동**: Supabase 마이그레이션을 SQL 파일이 아닌 MCP로 직접 실행. Vercel 배포 상태를 CLI 없이 MCP로 확인.

### 한계와 극복 방법
1. **AI가 한국 교육과정을 정확히 몰랐음**: Claude가 성취기준을 무시하고 자체 지식으로 마인드맵을 생성. → 성취기준을 프롬프트에 명시적으로 주입하고, schema 구조로 출력 형태를 강제하여 해결.
2. **대화 세션 한계**: Claude Code CLI는 세션 단위로 동작하여 24시간 무중단이 불가능. → Memory 시스템으로 세션 간 연속성 유지, 태스크 목록으로 진행 상황 추적.

## 6. 성과 요약

| 지표 | 수치 |
|------|------|
| 총 개발 시간 | ~30시간 (Day 1-2 집중) |
| AI 활용 비율 | ~90% (코드 생성, 디버깅, 문서 모두 포함) |
| AI로 절약한 시간 (추정) | ~120시간 (동일 기능을 혼자 구현 시) |
| 총 커밋 수 | 11개 (구현 시작 이후) |
| 총 페이지 수 | 17개 라우트 |
| AI API 엔드포인트 | 5개 |
| Supabase 테이블 | 6개 (모두 RLS 적용) |
| Vercel 배포 | 8회 자동 배포 |

### AI 없이는 불가능했을 기능
1. **기억의 궁전 기억 스토리 생성**: 각 개념을 장소 특성과 결합하여 VAKOG 다감각 스토리를 자동 생성하는 것은 인간이 수동으로 하면 단원당 2-3시간이 필요하지만, AI는 30초 만에 생성.
2. **교과서 구조 마인드맵**: 성취기준에서 소단원→세부주제까지 계층적으로 분해하여 정확한 정의/과정/예시를 포함한 마인드맵을 자동 생성.
3. **소크라틱 수행평가 피드백**: 답을 직접 주지 않고 질문으로 유도하는 교육학적 피드백을 루브릭 기반으로 자동 생성.

### 핵심 교훈
> AI는 "코드를 대신 짜주는 도구"가 아니라, "역할이 분리된 전문 팀"으로 구성할 때 가장 강력하다. 솔로 개발자가 AI를 팀으로 만들면, 기획→리서치→설계→구현→테스트→배포의 전체 파이프라인을 혼자 운영할 수 있다. 핵심은 AI에게 "무엇을 만들어줘"가 아니라 "이 역할을 맡아줘"라고 지시하는 것이다.
